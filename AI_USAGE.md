# AI_USAGE

## 1. 사용한 AI 도구

- **OpenAI Codex**
- **Claude Code** (Anthropic)

> 모델 버전은 세션/시점에 따라 변동될 수 있어, 본 문서는 실제 작업에 사용한 도구 단위로 기록했습니다.

### 도구별 역할 분담

| 도구 | 주요 사용 단계 | 선택 이유 |
|------|--------------|---------|
| Codex | 설계 초안 작성, 기능 구현, 리팩토링 전반 | 토큰 소모가 적어 반복적인 코드 생성·수정 작업에 적합. 빠른 응답으로 구현 루프를 빠르게 돌릴 수 있었음 |
| Claude Code | 설계 검증, FSD 레이어 준수 검토, 요구사항 전수 검증 | Codex 대비 코드 품질과 추론 정확도가 높아 "이게 맞는 방향인가"를 판단해야 하는 검증 단계에 적합 |

**운영 방식**: Codex로 설계 초안과 구현을 빠르게 진행하고 → Claude Code로 설계 정합성·FSD 레이어·요구사항 누락 여부를 검증한 뒤 → 수정이 필요한 부분을 다시 Codex로 반영하는 흐름으로 진행했습니다.

> **AI 결과물 수용/거부 기준**: 기술적 정확성뿐 아니라 ① 실무 관점의 자연스러움, ② 과제 스펙 해석의 여지, ③ 사용자 경험 직관성 세 가지를 기준으로 판단했습니다. AI 제안이 기술적으로 올바르더라도 위 기준에 맞지 않는다고 판단되면 직접 수정하거나 제안 범위를 확장했습니다.

## 2. AI 활용 방식

단순히 AI에게 질문을 던지는 방식 대신, **워크플로우를 설계하고 그 흐름대로 AI를 도구로 사용**하는 방식을 택했습니다. 구체적으로는 AI가 절대 건드려선 안 되는 규칙을 먼저 정의하고, 단계별로 AI에게 맡길 범위와 직접 판단할 범위를 분리한 뒤 순서대로 실행했습니다.

---

### 2.0 절대 규칙 (AI가 임의로 변경 불가한 항목)

워크플로우 시작 전, 아래 규칙을 컨텍스트에 명시하고 어떤 구현 단계에서도 이를 위반하면 반드시 재작업하도록 제약했습니다.

| 규칙                                                                      | 이유                                  |
| ------------------------------------------------------------------------- | ------------------------------------- |
| `db.json` 원본 파일은 절대 수정 불가                                      | 과제 명세의 명시적 제약 조건          |
| FSD 레이어 의존 방향(`app → widgets → features → entities → shared`) 준수 | 레이어 경계 붕괴 시 확장성 상실       |
| `any` 타입 사용 금지                                                      | TypeScript 도입 목적 훼손             |
| `import`로 데이터를 직접 로딩하는 방식 금지 (반드시 fetch 비동기 통신 사용) | 과제 명세의 명시적 제약 조건          |
| 비즈니스 로직(필터·계산·정규화)을 UI 컴포넌트 내부에 포함 금지            | 관심사 분리 원칙 유지                 |
| 필수 요구사항(3.1~3.4) 항목 임의 생략 금지                                | 평가 기준 항목이므로 반드시 전부 구현 |

---

### 2.1 AI 워크플로우 단계

단계별로 AI에게 맡긴 범위와 직접 판단한 범위를 구분하며 진행했습니다.

```
[1단계] 요구사항 분석                               
  입력: ORIGINAL.md 전문
  AI 역할: 필수/선택 항목 체크리스트 생성, 평가 포인트 항목 식별
  직접 판단: 구현 우선순위 결정
      ↓
[2단계] 아키텍처 설계                              
  입력: 체크리스트 + db.json 스키마
  AI 역할: FSD 폴더 구조 초안 생성, 데이터 파이프라인 흐름 초안 제안
  직접 판단: 레이어 의존 방향 확정, 각 단계의 책임 경계 결정
      ↓
[3단계] 데이터 전처리 전략                        
  입력: db.json 원본 (직접 탐색하여 이상 케이스 목록화)
  AI 역할: normalize 함수 구현 제안
  직접 판단: 이상 케이스 발굴, 처리 방향(drop vs fallback) 결정
      ↓
[4단계] 기능 구현                             
  입력: 설계 문서 + 절대 규칙
  AI 역할: 각 기능 단위 코드 구현
  직접 판단: 평가 포인트 항목 로직 직접 검증, UX 판단이 필요한 동작 수정
      ↓
[5단계] 공통 컴포넌트 추출 및 CSS 구조 정리       
  입력: 구현된 전체 코드
  AI 역할: CSS 파일 분리 제안
  직접 판단: 컴포넌트 추출 범위 확장 결정 (AI 제안 범위를 직접 확대)
      ↓
[6단계] 요구사항 전수 검증                       
  입력: ORIGINAL.md + 전체 구현 코드
  AI 역할: 누락 항목 체크리스트 작성
  직접 판단: 누락 여부 최종 판단 및 추가 구현 결정
```

### 2.2 주요 작업 단위

| 작업                 | 내용                                                                                | 직접 개입 포인트                                                                        |
| -------------------- | ----------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------- |
| 요구사항 분석        | ORIGINAL.md 기준으로 필수/선택 항목을 체크리스트로 구조화, 누락 없이 구현 범위 확정 | AI 체크리스트를 기반으로 구현 우선순위와 선택 항목 범위는 직접 결정                     |
| 아키텍처 설계        | FSD(Feature-Sliced Design) 기반 폴더 구조 설계 및 레이어 간 의존 방향 결정          | AI 초안을 기반으로 레이어 의존 방향(app→widgets→features→entities→shared) 확정 및 경계 조정 |
| 데이터 파이프라인    | fetch → normalize → merge → filter → aggregate → derive 흐름 설계                   | 각 단계의 책임 경계 기준은 직접 결정. AI는 흐름 구현만 담당                             |
| 예외 처리 전략       | db.json 내 이상 데이터(날짜 포맷 불일치, null, 중복 daily_stat) 처리 방안 수립      | 이상 케이스 발굴은 직접 db.json 탐색. AI는 처리 로직 구현만 제안                        |
| 전체 기능 구현       | 글로벌 필터, 일별 추이 차트, 캠페인 테이블, 등록 모달, 도넛 차트, 랭킹 Top3         | 평가 포인트 항목(도넛 연동, 일괄 상태 변경 등)의 로직은 직접 검증 및 수정               |
| UX 개선              | 날짜 역전 방지, 토스트 색상, 모달 가독성, 선택 배지 위치 등                         | 에러 메시지→자동 보정 전환 등 UX 판단은 AI 초안을 거부하고 직접 결정                    |
| 공통 UI 컴포넌트화   | raw HTML 태그를 Button/Input/Select/Chip/Badge로 추출, shared/ui 구성               | CSS 분리와 컴포넌트 추출 병행은 AI 제안 범위를 직접 확장                                |
| CSS 구조 정리        | index.css 단일 파일 → button.css/input.css/chip.css/toast.css 등 관심사별 분리      | CSS 분리만 진행하는 AI 제안에서 컴포넌트 추출 병행으로 방향을 직접 전환                 |
| 요구사항 검증        | 전체 구현 완료 후 ORIGINAL.md 기준으로 빠진 항목 재점검                             | 최종 누락 여부 판단 및 추가 구현 결정은 직접 수행. AI는 체크리스트 도구로만 활용        |

### 2.3 실제 프롬프트 예시

**설계 단계**

- "db.json에 포함된 날짜 포맷 불일치, null, 중복 데이터를 안전하게 처리하는 normalize 함수를 설계해줘"
- "글로벌 필터와 테이블 검색이 독립적으로 동작하도록 데이터 파이프라인을 설계해줘"
- "플랫폼 도넛 차트 클릭 시 글로벌 필터와 양방향 연동하고, 글로벌 필터 적용 시 다른 위젯과 수치가 동일하게 동기화되게 구현해줘"

**구현 단계**

- "글로벌 필터 날짜 from > to 역전 시 자동 보정 처리해줘"
- "일괄 상태 변경 후 토스트 피드백을 추가해줘"
- "SummaryStats에 있는 raw/dropped/merged 디버그 정보를 제거하고 의미 있는 지표만 보여줘"
- "캠페인 등록 모달의 시작일 기본값을 오늘로, 종료일을 오늘+7일로 설정해줘"

**리팩토링 단계**

- "button, input, select, chip 등 raw HTML 태그를 shared/ui 공통 컴포넌트로 추출해줘"
- "CSS가 index.css 하나에 뒤섞여 있는 문제를 관심사별로 분리해줘"
- "ORIGINAL.md 요구사항 기준으로 현재 구현에 빠진 항목을 전수 검증해줘"

**FSD 준수 검증 단계**

- "전체 import를 레이어별로 검토해서 FSD 위반 항목을 찾아줘"
- "shared가 entities를 참조하는 구조를 FSD에 맞게 재배치해줘"
- "app 레이어에 있는 UI와 훅 중 더 적합한 레이어가 있는 것들을 이동해줘"

### 2.4 AI 워크플로우를 이렇게 설계한 이유 (확장 설명)

이번 과제에서 AI는 "정답 생성기"가 아니라 "생산성 높은 구현 보조 도구"로 사용했습니다. 핵심은 AI가 코드를 빠르게 만들더라도, 요구사항 해석과 최종 책임은 사람이 갖는 구조를 의도적으로 만들었다는 점입니다. 이를 위해 작업을 작은 단위로 분해하고, 각 단위마다 입력 컨텍스트와 완료 조건을 먼저 정의한 뒤 AI에게 요청했습니다.

- 먼저 요청 단위를 작게 나눠 "무엇을 바꿔야 하는지"와 "무엇을 바꾸면 안 되는지"를 함께 전달했습니다. 예를 들어 필터 로직을 수정할 때는 UI 컴포넌트까지 건드리지 않도록 범위를 제한했고, 반대로 UI 개선 작업에서는 비즈니스 계산 함수를 수정하지 않도록 경계를 명확히 했습니다.
- 다음으로 프롬프트를 기능 설명 중심이 아니라 검증 가능한 조건 중심으로 작성했습니다. "도넛 차트 구현"처럼 넓은 요청보다 "글로벌 필터와 수치가 불일치하지 않도록 동일 필터 결과를 사용"처럼 테스트 가능한 조건을 우선 제시해 결과물의 품질 편차를 줄였습니다.
- 생성된 결과는 바로 수용하지 않고, 요구사항 원문(ORIGINAL.md)과 실제 동작을 기준으로 교차 검증했습니다. 특히 해석 여지가 있는 문구(기간 필터, 적용/해제 의미, 시작일/종료일 관계)는 문서 문구와 코드 동작이 어긋나지 않도록 최종 기준을 고정했습니다.
- 마지막으로 "문서와 구현의 정합성"을 별도 검증 항목으로 관리했습니다. 구현만 맞고 문서가 틀리면 면접/평가에서 신뢰도가 떨어질 수 있기 때문에 README와 AI_USAGE를 코드와 함께 동기화하는 과정을 반복했습니다.

이 방식의 장점은 두 가지였습니다. 첫째, AI가 만든 코드의 속도 이점을 유지하면서도 과제 해석 리스크를 낮출 수 있었습니다. 둘째, 기능 구현이 끝난 뒤에도 변경 이력과 의사결정 근거를 설명 가능한 형태로 남길 수 있어, 면접 단계의 기술 질의응답 대응력이 좋아졌습니다.

### 2.5 품질 게이트 운영 방식

워크플로우의 마지막 단계에서는 "작동한다" 수준이 아니라 "요구사항과 일치한다" 수준으로 확인하기 위해 품질 게이트를 고정했습니다.

- 코드 게이트: 타입 오류, 린트 경고, 단위 테스트 실패를 먼저 제거
- 요구사항 게이트: ORIGINAL.md 필수/선택 항목 체크리스트와 구현 기능을 항목별 대조
- 정합성 게이트: README/AI_USAGE 설명 문구가 실제 코드 동작과 일치하는지 확인
- UX 게이트: 토스트 피드백, 모달 접근성(포커스 관리/ESC 종료), 폼 재진입 시 초기화 등 사용자 흐름 기준 확인

특히 이번 과제에서는 "로직 정답"뿐 아니라 "해석의 일관성"을 중요하게 다뤘습니다. 예를 들어 기간 필터 해석이나 종료일 유효성처럼 문구 해석이 갈릴 수 있는 부분은, 코드·테스트·문서를 한 세트로 맞추어 변경했습니다. 이 과정을 통해 AI 출력의 흔한 문제인 문서/코드 불일치를 줄이고, 결과물의 설명 가능성을 높였습니다.

### 2.6 AI 입력 컨텍스트 파일

과제 가이드의 "컨텍스트 파일 기술" 요구에 맞춰, 실제로 AI 요청 시 함께 참고하거나 검증에 사용한 파일과 목적을 아래처럼 관리했습니다.

| 파일 | 사용 목적 |
|------|----------|
| `docs/ORIGINAL.md` | 필수/선택 요구사항 원문 기준 확보, 해석 충돌 항목(기간 필터/적용-해제/종료일 규칙) 검증. 과제 원문이라 git에는 포함되지 않습니다. |
| `docs/DESIGN.md` | 아키텍처 초안과 구현 결과 비교, 데이터 흐름/레이어 책임 일치 여부 확인 |
| `docs/DATA_SANITIZATION_PLAN.md` | 예외 데이터 처리 기준(정규화/드롭/보정)의 일관성 검증 |
| `docs/ASSIGNMENT_PLAN.md` | 단계별 진행 상태 추적, 누락 요구사항 재점검 |
| `README.md` | 구현 설명 문서와 실제 코드 동작의 정합성 검토 |
| `src/**` 주요 구현 파일 | 필터/집계/유효성/상태 관리 로직의 실제 동작 확인 |

문서 작성 단계에서는 "코드가 맞는가"와 "문서가 코드를 정확히 설명하는가"를 분리해 검토했습니다. 이를 통해 구현 완료 이후에도 README와 AI_USAGE가 실제 동작과 어긋나지 않도록 반복 보정했습니다.

## 3. 의사결정 과정 (대안 비교)

### 상태 관리 방식

- **대안 A**: React local state 중심
- **대안 B**: Zustand(클라이언트 전역) + TanStack Query(서버 상태) 분리
- **선택**: B
- **이유**: 글로벌 필터 동기화, 로컬 캠페인 등록, 상태 override를 단일 스토어로 관리하고, API 캐싱/로딩/에러를 Query가 담당하면 관심사가 명확히 분리됨

### Mock API 방식

- **대안 A**: MSW(Mock Service Worker) + fetch 비동기 통신
- **대안 B**: json-server + fetch 비동기 통신
- **선택**: B
- **이유**: 과제 제약 조건(import 방식 금지, 비동기 통신 필수)을 둘 다 충족하지만, json-server는 실제 HTTP 서버를 띄워 네트워크 레이어를 그대로 사용하므로 "API 서버와 통신하는 환경"을 더 충실히 재현할 수 있음. MSW는 Service Worker 기반으로 브라우저 내부에서 요청을 인터셉트하는 방식이라 별도 프로세스 없이 편리하지만, 실제 서버 없이 네트워크 탭에 요청이 표시되지 않아 디버깅 가시성이 낮음. 과제 명세에서 `json-server`를 API 예시로 직접 명시하고 있다는 점도 선택의 근거가 됨

### 플랫폼 도넛 필터 처리

- **문제**: 도넛 토글 UX를 위해 platform 필터를 무시하면, 특정 기간(특히 1일 필터)에서 도넛과 다른 위젯 간 수치 불일치가 발생함
- **최종 결정**: `usePlatformPerformanceData`도 글로벌 필터(`dateRange`, `statuses`, `platforms`)를 동일하게 적용
- **근거**: 필수 요구사항 3.1의 "모든 차트와 테이블 실시간 동기화"를 우선 충족하는 것이 평가 리스크가 낮음

### 상태/매체 최소 선택 정책

- **결정**: 상태 필터와 매체 필터는 최소 1개 이상 선택 상태를 강제
- **의도**: 빈 선택으로 인한 "완전한 빈 대시보드" 상태를 방지하고, 필터 토글의 피드백을 유지
- **트레이드오프**: 과제 문구의 "적용/해제"를 엄격히 해석하면 마지막 1개 해제도 허용해야 한다는 해석이 가능하므로, 이 부분은 README에 명시하고 면접에서 설계 의도를 설명할 예정

### 공통 UI 컴포넌트 설계

- **문제**: 프로젝트 전반에 raw HTML 태그가 직접 사용되어 스타일 일관성 유지 어려움
- **결정**: `shared/ui`에 Button, Input, Select, Chip, Badge 추출
- **방식**: 기존 CSS 클래스 기반 스타일을 유지하면서 `variant` prop으로 변형 제어. forwardRef로 react-hook-form 호환성 확보

### 날짜 역전 처리 방식

- **대안 A**: 에러 메시지 표시
- **대안 B**: 자동 보정 (from > to 시 반대편 날짜를 맞춰줌) + `min` 속성 추가
- **선택**: B
- **이유**: 브라우저 네이티브 date picker의 `min` 제약이 사용자에게 더 직관적이며, 역전 상태가 발생하지 않으면 에러 메시지 자체가 필요 없음

## 4. AI 결과물을 직접 수정·확정한 판단 사례

AI 초안을 검토한 뒤 직접 판단해 수정하거나 확정한 사례들입니다.

### 사례 1: 종료일 유효성 조건 확정

- **AI 초안**: `endDate >= startDate` (당일 포함)
- **직접 수정**: `endDate > startDate` (엄격 초과)
- **이유**: 과제 문구의 "시작일 이후"를 엄격 해석해 동일 시작/종료일을 허용하지 않도록 기준을 고정했고, 관련 테스트도 동일 기준으로 정합성을 맞춤

### 사례 2: 날짜 필터 역전 처리 방식

- **AI 초안**: from > to 시 에러 메시지 표시
- **직접 수정**: 조용한 자동 보정 + `min` 속성으로 브라우저 레벨 제약
- **이유**: 에러 메시지보다 아예 선택 불가 처리가 UX상 더 자연스러움

### 사례 3: SummaryStats 디버그 정보 제거

- **AI 초안**: `raw X / dropped Y / merged Z` 형태의 파이프라인 내부 통계를 사용자 화면에 표시
- **직접 수정**: 캠페인 수, 총 집행금액, CTR/CPC/ROAS 등 사용자에게 의미 있는 지표만 노출
- **이유**: 내부 디버그 정보는 실제 서비스 UI에 부적합하고, 평가자 입장에서도 "왜 UI에 있지?"라는 의문을 유발할 수 있음

### 사례 4: 모달 종료일 초기값 설계

- **AI 초안**: 빈 값으로 초기화
- **직접 수정**: 시작일 = 오늘, 종료일 = 오늘 + 7일로 기본값 설정
- **이유**: 사용자가 날짜를 직접 입력하는 번거로움 감소. 과제 스펙에 초기값 제약이 없으므로 UX 개선으로 자유롭게 추가 가능

### 사례 5: 선택 배지 위치

- **AI 초안**: 일괄 적용 버튼 옆에 "N개 선택됨" 배지 위치
- **직접 수정**: "캠페인 관리 테이블" 제목 옆으로 이동
- **이유**: 테이블 제목 옆이 선택 상태를 더 직관적으로 전달하며, 툴바 영역이 복잡해지는 것을 방지

### 사례 6: CSS + UI 컴포넌트 동시 분리

- **AI 초안**: CSS 파일만 관심사별로 분리
- **직접 확장**: CSS 분리와 함께 shared/ui 공통 컴포넌트(Button, Input, Select, Chip, Badge)도 동시에 추출
- **이유**: 스타일만 분리하고 컴포넌트는 raw HTML 태그를 유지하면 절반만 정리된 상태. 둘을 함께 해야 일관성 있는 구조가 됨

### 사례 7: FSD 레이어 위반 수정

- **AI 초안**: `shared/api/campaigns.ts`, `shared/api/dailyStats.ts`, `shared/lib/filter.ts`, `shared/lib/aggregate.ts`가 `entities` 레이어의 타입과 함수를 직접 참조하는 구조로 구현
- **직접 수정**: 전체 import를 레이어별로 재검토 후 위반 항목 식별
  - `shared/api/campaigns.ts` → `entities/campaign/api/campaigns.ts`
  - `shared/api/dailyStats.ts` → `entities/daily-stat/api/dailyStats.ts`
  - `shared/lib/filter.ts` → `entities/campaign/lib/filter.ts` + `entities/daily-stat/lib/filter.ts`로 분리
  - `shared/lib/aggregate.ts` → `entities/daily-stat/lib/aggregate.ts`
  - `app/ui/GlobalFilterBar.tsx` → `features/global-filter/ui/`
  - `app/ui/SummaryStatsSection.tsx` → `widgets/summary-stats/ui/`
  - `app/hooks/useGlobalFilter.ts` → `features/global-filter/model/`
  - `app/hooks/usePlatformPerformanceData.ts` → `widgets/platform-performance-donut/model/`
- **이유**: FSD에서 `shared`는 어떤 상위 레이어도 참조해서는 안 됨. `app`은 Provider·최상위 조합 역할만 해야 하며, 특정 feature·widget 전용 UI/훅은 해당 슬라이스 내부에 위치해야 함
