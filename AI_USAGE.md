# AI_USAGE

## 1. 사용한 AI 도구
- Codex (GPT-5 계열 코딩 에이전트)

사용 이유:
- 과제 요구사항을 체크리스트로 구조화
- README/설계 문서 템플릿을 빠르게 정리
- 요구사항 누락 여부 점검 자동화

## 2. AI 활용 방식
### 2.1 주요 작업 단위
- 과제 명세를 기능 단위(필수/선택)로 분해
- 제출 필수 문서(`README.md`, `AI_USAGE.md`) 충족 여부 점검
- 요구사항 기반 설계 문서(`DESIGN.md`) 작성

### 2.2 실제 프롬프트 예시
- "과제 가이드 기준으로 현재 README 누락 항목을 점검해줘"
- "요구사항에 맞는 아키텍처/컴포넌트 설계 문서를 작성해줘"
- "AI_USAGE.md 필수 항목(도구, 프롬프트, 의사결정, 직접 수정 사례)에 맞춰 정리해줘"

## 3. 의사결정 과정
- 기본 Vite README는 과제 제출 기준에 맞지 않아 전면 교체
- 구현 완료 상태가 아니므로 README에 "요구사항 반영 계획" 체크리스트를 분리
- 설계 근거를 면접 설명용으로 분리하기 위해 `DESIGN.md`를 별도 작성

## 4. 컨텍스트로 사용한 파일
- `README.md` (기존 상태 확인)
- `ASSIGNMENT_PLAN.md` (초기 설계 초안)
- `DATA_SANITIZATION_PLAN.md` (전처리 방향 참고)
- `package.json` (실행 명령/스택 근거)
- `src/*` (현재 구현 범위 확인)

## 5. AI 결과물을 직접 수정/판단한 사례
- 과제 요구사항은 "구현 참고 이미지 복제"가 아닌 "요구사항 충족"이 핵심이므로, 문서 구조를 기능 체크 중심으로 수정
- 구현되지 않은 기능을 완료처럼 보이게 작성하지 않고, 체크리스트 형태로 현재 상태를 명확히 표기
- 필수 문서 누락(`AI_USAGE.md`)을 확인 후 별도 파일로 추가

## 6. 한계와 보완
- AI가 제안한 설계는 프로젝트 초기 기준의 목표 구조이며, 실제 구현 과정에서 코드 단위 검증과 리팩터링이 필요함
- 최종 제출 전에는 기능 동작 캡처, 테스트 결과, 예외 케이스 재검증을 수동으로 수행할 예정

## 7. 프롬프트/의사결정/컨텍스트 기록 방식
### 7.1 사용한 프롬프트와 의도
- 프롬프트: "ORIGINAL.md 요구사항 기준으로 현재 구현 누락 항목을 체크리스트로 정리해줘"
- 의도: 필수/선택 요구사항을 빠르게 분해해 구현 우선순위를 잡기 위함
- 결과 활용: Day별 실행 계획(`ASSIGNMENT_PLAN.md`)으로 전환

- 프롬프트: "0분모/null/포맷 불일치가 있는 데이터 정규화 전략을 제안해줘"
- 의도: 지표 계산 안정성 확보
- 결과 활용: normalize/metrics 유틸 설계에 반영

- 프롬프트: "README를 과제 제출 기준(실행방법/스택근거/구조/설계)으로 재작성해줘"
- 의도: 제출 필수 문서 요건 충족
- 결과 활용: README 구조 개편 후 실제 구현 상태에 맞게 수동 수정

### 7.2 의사결정 과정(대안 비교)
- 이슈: 상태 관리 방식 선택
- 대안 A: React local state 중심
- 대안 B: Zustand + React Query 분리
- 선택: B
- 이유: 글로벌 필터 동기화와 비동기 데이터 캐시를 분리하면 차트/테이블 동기화가 단순해짐

- 이슈: mock API 방식
- 대안 A: import로 JSON 직접 로딩
- 대안 B: json-server API 통신
- 선택: B
- 이유: 과제 제약(import 금지, 비동기 통신 필수) 충족

### 7.3 AI에 제공한 컨텍스트 파일과 이유
- `ORIGINAL.md`: 과제 원문 기준(정답 기준) 확인
- `DESIGN.md`: 아키텍처/데이터 흐름 기준 유지
- `ASSIGNMENT_PLAN.md`: 4일 일정/우선순위 관리
- `DATA_SANITIZATION_PLAN.md`: 예외 데이터 처리 정책 일관성 확보
- `src/*`: 실제 구현 상태와 문서 내용 정합성 검증

### 7.4 AI 결과물을 직접 수정한 사례
- 사례 1: 7일 플랜 제안 결과를 4일 플랜으로 재구성
- 이유: 남은 일정에 맞는 실행 가능성 확보

- 사례 2: 문서가 구현 완료처럼 보이던 표현을 체크리스트형으로 수정
- 이유: 실제 구현 상태와 문서 정합성 유지

- 사례 3: 차트 토글 로직에서 최소 1개 선택 유지 조건을 강제
- 이유: 원문 요구사항(최소 1개 선택 유지) 충족
